# MCP 游戏测试功能介绍与工作流策略规范

## 背景定位

现阶段，依赖通用 Agent 仅凭纯 LLM 推理、截图识别和自由操作来完成复杂游戏测试并不现实。游戏运行环境具有状态空间大、异步事件多、画面反馈不稳定、复现成本高等特点；如果让 Agent 像真人一样“看画面、猜状态、尝试点击”，往往会导致测试链路不可复现、统计结果不可量化，也难以在回归测试中稳定复用。

MCDK 的 MCP 能力更适合作为“开发期可观测测试通道”：在开发代码时预留明确的测试函数、调试入口和日志采样点，再通过 MCP 的客户端 / 服务端代码执行 Tool 触发这些入口，并持续收集日志与返回值做统计分析。这种方式比纯视觉自动化更稳定，也更符合复杂游戏逻辑的工程化验证需求。

## 核心原则

1. **不要把复杂游戏测试完全交给纯 LLM**：LLM 可以负责规划、生成测试脚本、分析日志和归纳异常，但不应承担全部真实世界交互与状态判定。
2. **在代码开发阶段预留测试入口**：为关键业务逻辑保留只在开发环境启用的测试函数、诊断命令或调试接口。
3. **优先使用可观测信号**：优先采集返回值、结构化日志、错误日志、计数器和状态快照；截图与点击仅用于少量必须人工视觉确认的场景。
4. **让 MCP 调用成为测试执行层**：通过 MCP Tool 执行客户端 / 服务端代码、重载游戏、查询日志、捕获画面，从而把 Agent 的职责从“盲测”转移为“调用、采样、统计、分析”。
5. **测试结果应可复现、可聚合、可回归**：每次测试都应输出可被解析的标记日志或结构化数据，便于多轮统计。

## 推荐工作流

### 1. 开发期设计测试函数

在业务代码中为关键功能保留测试函数，例如：

- 初始化固定场景；
- 构造指定实体、方块、UI 或物品状态；
- 触发一次目标逻辑；
- 输出结构化结果；
- 清理测试环境。

这些测试函数应尽量满足：

- **幂等**：多次调用不会污染环境，或会先清理旧状态；
- **短链路**：一次调用只验证一个明确目标；
- **结构化输出**：日志中包含稳定前缀、测试名、结果、耗时和关键指标；
- **开发环境隔离**：只在 MCDK / 调试世界 / 测试开关启用时暴露。

### 2. 通过 MCP 执行测试入口

使用 MCP 客户端调用 MCDK 提供的 Tool：

- `execute_code`：在客户端或服务端执行预留测试函数；
- `get_latest_logs`：读取最近日志，收集结构化测试结果；
- `get_latest_error_logs`：优先确认是否存在 Python stderr 或异常；
- `reload_game` / `reload_addon_and_game`：仅在热更新无法覆盖时使用；
- `capture_game_window`：只在日志无法判断或需要视觉确认时使用。

推荐执行顺序：

1. 先确认游戏已由 MCDK 启动，且 `mcp_server_config.enabled` 已开启；
2. 调用测试函数前读取错误日志作为基线；
3. 调用 `execute_code` 触发测试函数；
4. 等待业务逻辑完成后读取最新日志；
5. 按稳定前缀解析测试结果；
6. 必要时重复执行多轮，统计成功率、耗时分布和异常分布；
7. 最后只在必要时截图确认视觉效果。

### 3. 日志格式建议

建议测试函数输出单行 JSON 或接近 JSON 的结构化日志，并带有稳定前缀，例如：

```text
[MCDK_TEST] {"case":"spawn_entity","ok":true,"duration_ms":18,"entity_id":"...","metrics":{"count":1}}
```

字段建议：

| 字段 | 说明 |
| --- | --- |
| `case` | 测试用例名，必须稳定 |
| `ok` | 布尔结果 |
| `duration_ms` | 测试耗时 |
| `stage` | 可选，记录初始化、执行、校验、清理阶段 |
| `error` | 可选，失败原因 |
| `metrics` | 可选，统计指标 |

### 4. Agent 的合理职责边界

Agent 适合做：

- 根据代码结构生成测试计划；
- 编写或补齐测试函数；
- 通过 MCP 调用测试入口；
- 收集日志、错误日志和必要截图；
- 对多轮结果做统计和异常归因；
- 输出可复现的回归步骤。

Agent 不适合做：

- 长时间依赖画面猜测游戏内部状态；
- 在无测试入口的情况下尝试覆盖复杂业务路径；
- 无限制点击、等待、重载或反复截图；
- 把一次偶然画面状态当作确定测试结论。

## 测试函数设计模板

Python2 Mod 侧可以保留类似结构：

```python
# 仅示意：实际项目可放在 Debug/Test/Diagnostics 模块中

def __mcdk_test_spawn_entity():
    import json
    import time

    start = time.time()
    result = {
        "case": "spawn_entity",
        "ok": False,
        "stage": "init",
        "duration_ms": 0,
        "metrics": {}
    }

    try:
        # 1. 清理旧状态
        # 2. 构造固定场景
        # 3. 触发目标逻辑
        # 4. 校验游戏状态
        result["stage"] = "assert"
        result["metrics"]["count"] = 1
        result["ok"] = True
    except Exception as exc:
        result["error"] = repr(exc)
    finally:
        result["duration_ms"] = int((time.time() - start) * 1000)
        print("[MCDK_TEST] " + json.dumps(result, sort_keys=True))

__mcdk_test_spawn_entity()
```

MCP 调用时只负责执行入口和收集结果，不应把大量业务判断写在临时执行字符串中。临时代码越短，越容易复现和排查。

## 策略分级

| 场景 | 推荐策略 |
| --- | --- |
| 纯逻辑函数 | 直接调用测试函数并解析返回 / 日志 |
| 客户端 UI 状态 | 调用 UI 测试入口输出控件状态，必要时截图 |
| 服务端玩法逻辑 | 服务端执行测试入口，读取服务端日志与错误日志 |
| 资源变更 | 优先热更新；失败后再 `reload_addon_and_game` |
| 视觉效果 / Shader | 先重载目标资源，再截图人工或模型辅助确认 |
| 长链路交互 | 拆成多个短测试入口，逐段统计 |

## 与 MCDK MCP 工具的关系

MCDK 已提供日志查询、错误日志查询、代码执行、重载、截图和点击等 MCP Tool。该规范强调：这些 Tool 不是为了让 Agent 模拟完整玩家，而是为了把游戏内部可测试能力暴露给外部自动化系统。最推荐的路径是：

```text
开发者预留测试函数 -> MCP execute_code 调用 -> 游戏内输出结构化日志 -> MCP get_latest_logs 收集 -> Agent 统计分析
```

## 结论

复杂游戏测试应采用“代码内测试入口 + MCP 执行通道 + 日志统计分析”的工程化策略。通用 Agent 与 LLM 的价值在于规划、调用、解析和归纳，而不是在无稳定观测点的情况下承担全部真实游戏测试行为。
